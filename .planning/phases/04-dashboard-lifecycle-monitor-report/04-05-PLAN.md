<objective>
Fix the broken DRAFT→PAPER_LIVE activation chain (the phase's most important control) AND generalize the activation pipeline's three seams so Phase 6 (real LIVE via Toss) is purely ADDITIVE.

Currently `PaperLifecycleService.promote()` only flips `status='PAPER_LIVE'` and never populates `paper_live_symbols`, so `LiveDataScheduler` Guard 3 (`activeSymbolsUnion()`) finds an empty table and skips every tick — no ingestion, no evaluation, no trades. Additionally `LiveEvaluationService.resolveSymbols()` reads symbols from `rule.definition.universe` while the scheduler ingests from `paper_live_symbols`, so the two halves of the loop are derived from different sources.

On top of that fix, three places hardcode behavior that Phase 6 would otherwise have to rewrite. We generalize the SEAMS ONLY (no LIVE behavior is built now):
- SEAM 1 — status predicate (no hardcoded `"PAPER_LIVE"` string in the live filters).
- SEAM 2 — executor routing by rule mode (select the right `OrderExecutorPort` per rule).
- SEAM 3 — mode-agnostic lifecycle symbol-assignment (no `"PAPER_LIVE"` baked into the symbol helper).

Purpose: Make promote actually start live paper trading end-to-end with a single canonical symbol source, and leave clean extension points so Phase 6 adds LIVE with single-line / new-class edits rather than rewrites.
Output: A promoted rule populates `paper_live_symbols`, the scheduler tick runs, `LiveEvaluationService` evaluates exactly the assigned symbols using a per-rule-selected executor, and the live filter uses a shared `RuleStatus.isLiveActive` predicate.

**Symbol-source reconciliation decision (EXPLICIT):** `paper_live_symbols` is the SINGLE canonical source for both ingestion and evaluation. promote() resolves the rule's symbols once (mirroring `BacktestService.resolveInitialSymbols`) and writes them to `paper_live_symbols`; `LiveEvaluationService.resolveSymbols()` is changed to READ from `paper_live_symbols` (via `PaperLiveSymbolRepository.findByRuleId`) instead of parsing `rule.definition.universe`. This guarantees ingested symbols == evaluated symbols and enables `volume_top_n` live rules (the gap flagged in 03-VERIFICATION "notable").

**Phase 6 boundary (DO NOT build now):** No LIVE order execution, no TossOrderExecutor, no TossLiveIntradayAdapter, no LIVE promotion gate, no circuit breaker. Market-data source stays unified for PAPER only — LIVE will use a SEPARATE `TossLiveIntradayAdapter` per TOSS-05 in Phase 6 (do NOT touch the market-data source seam here). LIVE promotion gating (Toss auth + 5 trading days, RULE-04) is Phase 6. We only make today's seams general enough that those additions are non-destructive.
</objective>
