# Phase 6.7: 실시간 거래량 상위 유니버스 (KRX 거래량 순위 연동) - Context

**Gathered:** 2026-06-22

**Source:** 대화 기반 결정 (insert-phase)

<domain>
## Phase Boundary

`volume_top_n` 룰이 "그날(현재 시점까지 누적) 거래량 상위 종목"을 **실시간으로 동적 선정**하도록 만든다. 현재 한계:
- **백테스트**만 날짜별 top-N을 적용하되, 기준이 `market_bars`의 **완결된 과거 일봉 거래량**(후행·일단위)이다 (`BacktestService.resolveSymbolsForDate` → `MarketBarRepository.findTopVolumeSymbolsOnDate`).
- **라이브**(PAPER_LIVE)는 top-N 랭킹을 **아예 적용하지 않고** 후보 전체(KOSPI200 ∪ additionalSymbols)를 평가한다 (`PaperLifecycleService.resolveSymbols` line 173–181이 후보 전체를 `paper_live_symbols`에 스냅샷 저장 → `LiveEvaluationService.resolveSymbols`가 그대로 사용).
- "지금 시점 횡단면 거래량 순위"를 한 번에 주는 **데이터 소스가 없다** (Yahoo는 per-symbol fetch라 종목을 알아야 호출 가능 → 전 종목 합산은 5분마다 ~200콜로 비현실적).

**아웃-스코프:** 실투자(LIVE) 주문(Phase 8), TradingView 연동(Phase 7), 신규 지표/연산자, KOSDAQ.
</domain>

<decisions>
## Implementation Decisions (locked — 사용자 확정)

### 1. 데이터 소스 = KRX MDC JSON
- data.krx.co.kr (MDC) 거래량/거래대금 순위 **JSON** 엔드포인트를 횡단면 소스로 도입한다. HTML 스크레이핑(네이버 sise_quant)이 아니라 JSON — 인코딩(EUC-KR)·레이아웃 변경·차단 리스크 회피, 견고성 우선.
- (research가 정확한 MDC 엔드포인트/파라미터/응답 스키마, intraday 갱신 주기, rate limit, 호출 방식을 확정한다. 네이버는 최후 fallback 후보로만 검토.)

### 2. 유니버스 범위 = 전체 KOSPI 보통주 − ETF/ETN
- KOSPI200 한정이 아니라 **전체 KOSPI 보통주** 중 거래량 상위 N.
- **ETF/ETN/우선주/관리종목/SPAC 제외** 필수 (이들이 거래량 상위를 독식). → `companies`에 instrument 구분(`instrument_type` 또는 `is_etf`) 컬럼 추가 또는 ETF/ETN denylist 유지. (research/plan에서 KRX 종목구분 메타로 분류 가능한지 확인.)

### 3. 추상화 = VolumeRankingProvider 포트
- 기존 `MarketDataPort`(`DbMarketDataAdapter`) 패턴을 따라 `VolumeRankingProvider` 포트 도입: `topVolume(market, asOf, topN, excludeEtf)`.
- 백테스트 impl = 기존 DB(`findTopVolumeSymbolsOnDate`). 라이브 impl = KRX 어댑터(~1분 캐시).

### 4. 라이브 동작 = 주기적 재선정 (승격 시 고정 X)
- 현재 `assignSymbols` 스냅샷 고정 대신, 라이브 volume_top_n은 **주기적으로 재선정**(매 틱 또는 N분)한다.
- ⚠️ **핵심 함정 (locked)**: top-N에서 이탈한 종목에 **보유 포지션이 있으면 청산 평가는 계속 수행**한다. 진입=신규 top-N, 청산=보유종목 포함. `paper_live_symbols` / `LiveEvaluationService.resolveSymbols` / `LiveDataScheduler.collectLiveData` / `PaperLiveSymbolService.activeSymbolsUnion`가 영향권.

### 5. 백테스트 ↔ 라이브 정의 불일치 처리
- 백테스트=과거 일봉 거래량, 라이브=당일 인트라데이 누적 거래량으로 기준이 다르다. 설계에서 **명시·문서화하고 일관성 방침을 결정**한다 (예: 차이를 수용하고 문서화 vs 백테스트도 인트라데이 누적 스냅샷 기준으로 정렬).

### Claude's Discretion
- KRX MDC 정확한 엔드포인트/파라미터/스키마, 캐시 TTL, 재선정 주기(매 틱 vs N분), 실패 시 fallback 정책
- ETF/ETN 분류 구현(컬럼 vs denylist), companies 시드 보강 범위
- 재선정 시 진입/청산 종목 집합 분리의 구체 구현(스케줄러 수집 대상 = 신규 top-N ∪ 보유종목)
- 백테스트/라이브 일관성 방침의 최종 선택
