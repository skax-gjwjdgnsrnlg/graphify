## Phase Boundary

두 가지를 전달한다:

1. **역할 분리** — 모의 룰 **설정** 화면(CRUD + DRAFT↔ACTIVE 관리)과 룰 **라이프사이클** 화면(ACTIVE 룰만 노출 + PAPER_LIVE 시작/중지 제어)의 역할을 명확히 분리한다. 두 화면이 동일 상태를 중복 표시하지 않는다.
2. **매매 근거** — 백테스트·모의(PAPER_LIVE) 매매 이력의 각 매수/매도에 어떤 조건(트리거 조건식 + 그 시점 지표값)으로 발생했는지 근거를 표시한다.

**인-스코프:** PaperRulesPage(설정), TradingRulesPage(라이프사이클) 역할 재정비, 2축 상태 모델 도입, RuleEvaluator 트리거 근거 계측, 백테스트/모의 이력 테이블 근거 확장 행
**아웃-스코프:** 새 지표/연산자 추가 없음, 룰 빌더 UI 변경 없음(Phase 6 완료), LIVE 승격/토스 연동 없음(Phase 8), 캔들 차트 없음(Phase 6.6)

<decisions>
## Implementation Decisions

### 1. ACTIVE 상태 모델 — 2축 분리
- 단일 `status` 문자열을 **config축 × run축** 2축으로 분리한다:
  - `config_status`: `DRAFT` | `ACTIVE`
  - `run_status`: `STOPPED` | `RUNNING` (RUNNING = 기존 PAPER_LIVE 평가 루프 포함)
- **DRAFT → ACTIVE 전환**: 조건 없이 사용자가 수동으로 올린다. 백테스트 불필요.
- **RULE-01 백테스트 게이트 폐지**: 기존 "PAPER_LIVE 승격 전 백테스트 1회 필수" 게이트를 **제거**한다. 백테스트 없이도 시작(RUNNING) 가능. (RULE-01에서 명시적으로 벗어나는 결정 — 사용자 승인)
- **ACTIVE 룰 편집**: STOPPED 상태면 자유롭게 편집 가능. **RUNNING 중에는 편집 차단**(실행 중 정의 변경 방지).
- **중지 동작**: RUNNING 룰을 중지하면 `ACTIVE / STOPPED`로 복귀(라이프사이클에 계속 남아 재시작 가능). 별도 PAUSED 상태로 빠지지 않음.
- **ACTIVE → DRAFT 하향**: RUNNING 중에는 차단(편집 규칙과 동일). 먼저 라이프사이클에서 중지해야 가능.

### 2. 두 화면 역할 경계 & 네비게이션
- **기존 2라우트 유지 + 재정비** (한 페이지 탭 통합 아님):
  - `/paper/rules` → **"전략 설정"** (PaperRulesPage): 룰 CRUD + DRAFT↔ACTIVE 토글
  - `/paper/rules-lifecycle` → **"전략 운영"** (TradingRulesPage): ACTIVE 룰만 + 시작/중지
- **사이드바/메뉴 이름**: `전략 설정` / `전략 운영`
- **전략 설정 화면 (정의 중심)**: 모든 상태 룰 나열. 컬럼 = 이름 · config 상태 배지(DRAFT/ACTIVE) · 쿨다운 · 수정일 · [편집 / 복제 / 삭제 / ACTIVE 토글]. **run 제어(시작/중지) 미노출.**
- **전략 운영 화면 (운영 중심)**: ACTIVE 룰만 표시. 컬럼 = 이름 · run 배지(실행 중 / 중지됨) · 마지막 신호 시각 · [시작 / 중지]. DRAFT 룰은 표시하지 않음.
- **중복 제거**: 설정은 config축만, 운영은 run축만 제어 → 동일 상태 중복 표시 없음 (SC3 충족).

### 3. 매매 근거 내용 수준
- **근거 = 조건식 + 지표값 둘 다**: 예) `진입: RSI(14)=28.3 < 30 ✓`. 충족된 조건식과 그 시점 실제 지표값을 함께 표시.
- **청산 사유 구분**: `익절(+X%)` / `손절(-X%)` / `지표 조건`을 구분해 표시. 현재 `exitTriggered`가 셋을 boolean으로 합치므로, **어떤 경로로 청산됐는지 reason을 반환하도록 재구성** 필요.
- **다중 조건(AND/OR)**: 그룹 내 **모든 조건을 각각 값 + 통과 여부(✓/✗)** 로 저장·표시. 조건별 평가 결과를 보존해야 함(그룹 boolean만으로 부족).
- **백테스트 = 모의 동일 형식**: 동일 근거 구조/컴포넌트 재사용. 백테스트 엔진도 모의의 `indicatorSnapshot`과 같은 JSON 스키마로 근거를 생성.

### 4. 매매 근거 표시 방식
- **행 클릭 → 확장(아코디언)**: 거래 행 클릭 시 아래로 근거 상세(조건별 값 + ✓/✗) 펼쳐짐.
- **적용 화면**: 백테스트 결과 거래 테이블(PaperBacktestPage) **+** 모의 거래 이력(PaperHistoryPage) **둘 다** (SC4 + SC5).

### Claude's Discretion
- 2축 상태의 DB 표현 방식(컬럼 추가 마이그레이션 vs 기존 status 문자열 매핑) — research/plan에서 결정
- 확장 행 상세 레이아웃·타이포그래피·✓/✗ 색상
- 근거 JSON 스키마 정확한 필드 구성
- 백테스트 trade ↔ 근거 데이터 연결 방식(인라인 vs 별도 리스트 인덱스 매칭)
- 모의 trade ↔ paper_signal_log 조인 키(rule+symbol+ts) 세부
